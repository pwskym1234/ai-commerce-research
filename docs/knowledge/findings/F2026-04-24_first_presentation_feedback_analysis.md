# 1차 발표 피드백 분석 — 아직 반영되지 않은 참고점

## 0. 분류 체계

| 색 | 상태 |
|----|------|
| ✅ 이미 반영 | 현 설계에 통합됨 |
| 🟡 부분 반영 | 일부만 반영, 강화 가능 |
| 🔴 미반영 | 신규 고려 필요 |

## 1. 이미 반영된 주요 피드백 (확인용)

| 피드백 | 반영 방법 |
|-------|---------|
| Position bias | R4 매 반복 셔플 (runner.py) |
| 재현성 (seed, model version, 반복) | R1~R5, jsonl 메타 |
| 단일 LLM 한계 | Phase 4 sanity check 계획 |
| 응답 일관성 (hallucination) | 20회 반복 + Wilson CI |
| 종속변수 이진/연속 | Y2a 5분화 |
| 통제 실험 외적 타당성 | Phase 4 + observational queries |
| 창업 vs 수업 톤 혼동 | 마스터 §9.1 분리 톤 명시 |

## 2. 🔴 미반영 — 신규 고려 필요

### 2.1 ★★★ 개인화(메모리·사용내역) 반영 부재 (복수 피드백)

> "AI가 이미 나에 대한 충분한 정보(메모리, 사용 내역)를 갖고 있고, 이를 바탕으로 추천을 하기 때문이라고 생각하는데, 이런 부분을 어떻게 반영할 수 있을지"
> "인공지능 서비스에게 쇼핑 검색을 하는 사람은 높은 확률로 유료 서비스를 구독하고 있을 것"
> "특정 사용자에게 상품이 등장할 확률이 높아지는 것을 검정할 수 있어도, 일반적인 AI 쇼핑을 하는 사람에게 추천 확률을 높이는 방법을 어떻게 검정할지"

**현 상태**: 우리 본실험은 **cold user** 시나리오 (세션 메모리 없음). 모델 기본 지식에 의존.

**반영 옵션**:
- **EXPLORATORY_3 신설**: "페르소나 컨텍스트를 system prompt에 주입할 때 F 효과가 달라지는가"
- 3가지 페르소나 샘플:
  - "나는 출산 2개월차, 30대 여성"
  - "나는 1인 가구 50대 남성, 전립선 수술 2년차"
  - "나는 갱년기 이후 골반저근 관리하는 50대 여성"
- 페르소나 × F 교호작용 검정 (H15 확장)
- 비용 추가: 페르소나 3 × 기존 8,640 = 25,920 추가 호출 = $26 더

**우선순위**: 🟡 P1. 시간 여유 있으면 Phase 4 전 추가.

### 2.2 ★★ "섹션 구분 명확성"이 F 요인에 없음

> "상품 정보의 각 세션이 명확하게 구분되어 있고 그래서 각 세션에서의 정보가 충분히 수집될 때 추천확률도 올라갈 것이다"

**현 상태**: F1(HTML 포맷 TABLE/BULLET/PARAGRAPH)이 구조화 정도를 일부 커버하지만, **섹션 간 의미적 구분**(스펙/리뷰/FAQ/비교 각각을 `<section>` 태그 + 명확한 `<h2>` 헤딩으로 분리)은 별도 차원.

**반영 옵션**:
- 가상 페이지 템플릿 (현재 모든 페이지 공통)에 **섹션 헤딩 명확성 고정 수준**으로 이미 반영됨 (section.block + h2 구조)
- 신규 F7 (섹션 명확성: 고/중/저) 고려 → L54에 차원 추가 시 L108 필요 → 비용 2배
- **권고**: F로 추가 안 함. 대신 **데마 관찰 변수**로 `section_count / text_length` 비율 분석 (이미 feature로 수집 중)

**우선순위**: 🟢 P2. 데마 변수로 충분.

### 2.3 ★★ SKU 매칭 / 동일 상품 라벨 다름 (데마 가공 이슈)

> "실제 쇼핑몰을 살펴보면 같은 제품도 여러 판매사에서 제공하고 있는 점을 알 수 있습니다. 그들마다 붙어있는 라벨이 다른데, 이를 하나로 묶는 방법이 필요"

**현 상태**: 현재 우리는 `brand × channel × sku_id` 단위로만 feature 집계. 같은 제품이 GN코스몰·쿠팡·네이버 각각 다른 SKU면 별개 페이지로 처리됨.

**반영 옵션**:
- 데마 전처리에 **SKU matching** 추가 (예: 페어와이즈 텍스트 유사도, 이미지 해시 비교, 브랜드+모델명 정규화)
- "같은 제품"을 한 entity로 묶은 후 브랜드별 평균 feature 계산
- 이미 features.jsonl에 `brand`로 집계되므로 SKU 다름은 복수 row로 자연 처리됨. 다만 **같은 제품의 채널별 차이**(예: 바디닥터 gncosshop 59 vs 네이버 스마트스토어)를 분석 변수로 활용 가능

**우선순위**: 🟡 P1. 데마 분석 단계에 명시.

### 2.4 ★★ 이미지 감성 분석 / 이미지 특성 활용

> "이미지와 같은 비정형 데이터를 이미지의 개수와 같은 정형 데이터로 변환하지 말고, 이미지 감성 분석등을 활용하면 더 좋은 효능을 가진 모델을 만들 수 있을 것 같습니다"

**현 상태**: `image_count` (정수)만 사용. 이미지 내용 미활용.

**반영 옵션**:
- GPT-5.4 vision 모드로 각 제품 hero 이미지 분석 (lifestyle/product-only/infographic 등 분류)
- CLIP embedding으로 이미지-텍스트 일관성 스코어
- **비용**: image 1장당 vision API ~$0.005 → 전체 SKU 이미지 수십 장 = 약 $1

**우선순위**: 🟢 P2. 데마 확장 과제. 본실험 범위 초과.

### 2.5 ★ SNS/커뮤니티 영향 반영 약함

> "sns에서의 노출 / 해시태그 / 댓글 분석"
> "SNS uses a mix of images and video and texts for promotion, so how will you check to see if AI really scans through SNS"

**현 상태**: H14 (외부 증거)에 리뷰·언급 포함이지만 **SNS 구체 지표 없음**. 네이버 카페 언급 빈도, Reddit 언급, 블로그 리뷰 수 등 미수집.

**반영 옵션**:
- consulting/action_roadmap.md Stage 4 "External Evidence"에 **SNS/해시태그 측정 구체화**:
  - 네이버 블로그 월별 언급 수
  - 맘카페 게시글 수
  - 유튜브 리뷰 영상 수
  - 인스타그램 해시태그
- 이 데이터를 데마 H14 관찰 변수로 추가

**우선순위**: 🟡 P1. 컨설팅 실행 단계에서 중요.

### 2.6 ★ "프롬프트 셧다운/API 제한 위험" 대응 명시 부족

> "GPT한테 자동으로 질문을 먹이면서 진행한다고 하셨는데 과도하게 프로그램이 돌아가서 셧다운 되거나 문제가 일어날 가능성은 없나요?"

**현 상태**: runner.py에 명시적 rate-limit backoff 없음. OpenAI client 기본 retry에 의존.

**반영 옵션**:
- runner.py에 **tenacity 기반 exponential backoff** 추가
- Rate limit 걸리면 자동 sleep 후 재시도
- 중간 실패 시 jsonl append로 재개 가능 (이미 구현)

**우선순위**: 🟡 P1. 본실험 직전 필수.

### 2.7 ★ "GEO·AiEO 시장 포화 우려 / 차별화"

> "GEO 시장은 신흥 시장은 아님. 이미 미국 시장은 Profound가 다 먹어치웠고, 국내 시장도 현재 어느 정도 기업들이 많이 갖춰진 상황임"
> "현재 시스템 및 연구와 차별화할 수 있는 점을 한 번 더 생각해주시면 좋을 것 같습니다"

**현 상태**: docs/trends_2026/APPLICATION.md + consulting/dashboard_upgrade_spec.md 에 차별화 방향 제시. 하지만 발표 슬라이드 수준 명시 부족.

**반영 옵션**:
- 발표 슬라이드 §1.5 추가: **"Profound·Alhena·체인시프트와의 차별화 4가지"**
  - (a) SKU-level (대부분 브랜드-level)
  - (b) 한국 이커머스 특화 (식약처 DB 연동)
  - (c) 통제 실험 기반 인과 추정 (대부분 관찰형)
  - (d) 의료기기·가글 버티컬 비교 분석

**우선순위**: 🟡 P1. 발표 준비 시 반드시 포함.

### 2.8 ★ "수업 프로젝트인가 창업 발표인가" 톤 혼동

> "개인의 창업 계획 발표인지 아니면 수업 팀프로젝트 발표인지 구분이 잘 안 갔습니다"

**현 상태**: CLAUDE.md §11에 "학술 구조 우선, 창업 비전은 질문 시만" 명시. 하지만 실제 1차 발표 구성에 혼동 남은 듯.

**반영 옵션**:
- 2차 발표 슬라이드 첫 3장을 **명시적 학술 구조**로:
  1. 연구 질문 (1장)
  2. 가설 H1~H15 (1장)
  3. 실험 설계 L54 + N=14 (1장)
- 창업 이야기는 맨 끝 "Implications" 1장만, 또는 질문 시만

**우선순위**: 🟡 P1. 발표 준비 필수.

### 2.9 🔴 "페이지 구성이 LLM 추천을 정말 바꿀 수 있는지 메커니즘 근거"

> "LLM의 추천 메커니즘은 (별도의 스폰서십이 없다고 가정할 때) 결국 사전 학습 데이터에 내재된 사용자 컨센서스, 즉 리뷰 수, 언급 빈도, 신뢰도 높은 소스에서의 인용 등에 기반할 가능성이 높다. 그렇다면 이 실험이 측정하는 것은 '페이지 구성이 추천에 미치는 진짜 영향'이 아니라, '통제된 프롬프트 환경에서 특정 표현 방식이 LLM의 즉각적인 출력에 미치는 영향'에 불과할 것 같다."

**현 상태**: 마스터 §1.3 클로즈드셋 정당화 있음. 하지만 이 피드백은 더 근본적 비판.

**반영 옵션**:
- 발표에 **"메커니즘 근거 슬라이드"** 추가:
  - Amazon Rufus 논문 (SPN 5 facet 기반)
  - ChatGPT Shopping card 트리거 조건 (Profound 리서치)
  - 우리 실험의 프레이밍: "실제 AI 쇼핑 에이전트가 상품 DB 제공 후 판정" = 클로즈드셋
- 이건 반복 피드백이라 **중요도 최고**

**우선순위**: 🔴 P0. 발표 성패 좌우.

### 2.10 🔴 완전요인설계 통계적 엄밀성

> "GPT 같은 생성형 AI는 'A,B,C,D 중 뭐 하나 추천해줘'라고 할 때 position bias로 인해 A를 추천할 가능성이 높은데 이를 해결할 방법?"
> "Phase 1.2는 연속형, Phase 2.2에서 제시된 평가지표는 범주형을 위한 것인데 이 간극을 해결할 수 있을지?"

**현 상태**: R4 셔플 ✅, Y2a 이진+연속 이중 정의 ✅. 해결됨.

→ **발표에서 이걸 반드시 설명**. Q&A 대비 슬라이드 준비.

### 2.11 🔴 "동일 상품이 여러 판매사에 있을 때 엔티티 매칭"

→ 2.3 참조. 중복 피드백.

## 3. 🟡 부분 반영

### 3.1 "특정 수요층 공략 vs 보편 질문"

> "인지도가 낮은 브랜드의 경우에는 특정한 수요층의 공략을 위한 그 제품만의 특징적인 부분을 내새우는 것이 더 효과적일 수 있을 거라고 생각하는데, 질문의 query가 보편적인 부분에만 한정된다면, 인지도 낮은 브랜드들에게 효과적인 모델이 될 수 있을까에 대한 의문"

**현 상태**: USE/SYM 쿼리에 일부 반영. 하지만 "니치 특화 vs 보편"의 명시적 대비 쿼리 없음.

**반영 옵션**: queries.yaml에 **"브랜드 인지도 × 쿼리 타입" 매트릭스** 명시. 예:
- 보편 (CAT "가글 추천") vs 니치 (USE "프로폴리스 계열 중 40대 여성 산후 회복용")
- 결과 분석 시 "니치 쿼리에서 바디닥터·프로폴린스 선택률이 보편보다 높은가" 검정

**우선순위**: 🟡 P1. 쿼리 2~3개 더 추가로 해결 가능.

### 3.2 "Tier 분류 프레임워크 / 시뮬레이션"

> "AI 인용 빈도와 크롤링 기재율을 교차 분석하여 변수별 Tier를 분류하는 EDA 방법론"
> "단순 예측을 넘어 시뮬레이션을 통해 점수가 낮은 페이지의 개선 방향을 제시하는 분석"

**현 상태**: consulting/action_roadmap.md에 Before/After 시뮬레이션 일부 반영. Tier 분류는 친구 대시보드 스타일로 정리 가능.

**반영 옵션**: 결과 정리 단계에서 F 요인 × 효과 크기를 **Tier 1/2/3**으로 분류 (Profound Asset Hierarchies와 유사).

**우선순위**: 🟡 P1. 결과 정리 단계.
