## 1. AiEO란 무엇인가

### 1.1 핵심 질문

> AI 에이전트가 상품을 추천할 때, 상품 페이지의 어떤 요소가 추천에 영향을 미치는가?

### 1.2 포지셔닝

- **GEO(Generative Engine Optimization)**: 콘텐츠/웹페이지 레벨에서 "AI 검색에 잘 걸리게" 최적화 → **발견(discovery)** 최적화
- **AiEO(AI Engine Optimization)**: SKU 단위에서 "AI 에이전트가 비교 평가할 때 우리 상품이 선택되게" 최적화 → **선택(selection)** 최적화
- 한 줄 요약: "GEO는 발견을 최적화하고, AiEO는 선택을 최적화한다."

### 1.3 실험 프레이밍

실제 AI 쇼핑 에이전트(Shopify Shop AI, Amazon Rufus, Perplexity Shopping 등)는 "세상 모든 상품을 떠올려서 추천"하는 게 아니라, **상품 DB/카탈로그를 API나 크롤링으로 받아온 다음 그 안에서 비교 평가**해서 추천한다. 즉 현실이 이미 클로즈드셋이다.

따라서 실험의 정의:
- ~~"LLM이 자유롭게 추천할 때 우리 제품이 나올 확률"~~
- → **"LLM이 상품 정보를 접했을 때 긍정적으로 평가할 확률"**

### 2.3 프로젝트에서 다루는 제품

- **메인 (산공통 통제 실험)**: 바디닥터 요실금치료기 (의료기기)

#### 선정 이유

의료기기는 AI 추천 시 고려 요소가 다양(인증, 스펙, 안전성, 임상 근거)해서 페이지 구조 최적화의 효과를 볼 수 있는 버티컬이고, 가글은 대표가 직접 요청 + 경쟁 제품이 풍부해서 데이터 수집이 용이함.

**핵심 전제**: 모든 버티컬이 동일한 AI 추천 로직을 가지지 않는다. 의료기기와 가글의 추천 드라이버가 다를 것이며, 이 차이 자체가 연구 발견이 된다.

## 4. 산업공학통계 프로젝트 상세

### 4.1 실험 설계: 분할 요인 실험 (의료기기 맞춤)

#### 조작 요인 (F1~F6, 의료기기 맞춤으로 재설계됨)

| 요인 | 수준 | 수준 수 | 유형 |
|------|------|--------|------|
| F1. HTML 포맷 | TABLE / BULLET / PARAGRAPH | 3 | 러닝화에서 유지 |
| F2. JSON-LD | Product 스키마 / MedicalDevice 스키마 / 없음 | 3 | 스키마 변경 |
| F3. 수치 구체성 | 정확("1200Hz 12단계") / 모호("강력한 EMS") | 2 | 러닝화에서 유지(축소) |
| F4. 인증 정보 위치·수준 | 상단+상세 / 하단+간략 / 없음 | 3 | 신규 (기존 F4 스펙위치 대체) |
| F5. 인증 명시 수준 | 인증번호+기관+등급 / "인증완료"만 / 없음 | 3 | 신규 (기존 F5 언어 삭제) |
| F6. 임상/효과 근거 | 임상 데이터 인용 / 사용자 후기 / 없음 | 3 | 신규 (기존 F6 URL구조 삭제) |

**전체 조합**: 3×3×2×3×3×3 = 486 → 직교 배열(Taguchi 등)로 **18~36개 조합**으로 압축.

**기존 대비 변경 요약**:
- F1(HTML 포맷), F3(수치 구체성): 유지. 범용적으로 유효한 요인.
- F2(JSON-LD): 유지하되 MedicalDevice 스키마 수준 추가.
- F4: "스펙 위치" → "인증 정보 위치·수준"으로 변경. 의료기기에서 인증이 AI 추천의 핵심 신호.
- F5: "언어" 삭제 → "인증 명시 수준" 신규. 한국 시장 중심이라 언어 요인 불필요.
- F6: "URL 구조" 삭제 → "임상/효과 근거" 신규. 프롬프트 주입 실험에서 URL 영향 미미.

#### 쿼리 수준 요인 (Q1~Q2)

| 요인 | 수준 |
|------|------|
| Q1. 쿼리 유형 | 정량("1200Hz 이상 케겔운동기") / 정성("출산 후 골반근육 회복") / 비교("케겔운동기 브랜드별 비교") / DECISION("케겔운동기 하나만 골라줘") |
| Q2. 조건 복잡도 | 1개 / 2개 / 3개 조건 |

#### 플랫폼 요인

| 요인 | 수준 |
|------|------|
| P1. AI 모델 | ChatGPT / Gemini / Perplexity |

### 4.2 종속변수 (Y1~Y4)

| 변수 | 의미 | 레이어 |
|------|------|--------|
| Y1. 파싱 정확도 | AI가 스펙 수치(Hz, 단계 수 등)를 정확히 읽었는가 | GEO |
| **Y2. 추천 선택률** | **경쟁 상품 중 우리 제품이 선택됐는가** | **AiEO 핵심** |
| Y3. 근거 수치 기반도 | 추천 이유가 스펙/인증 기반인가 감정 기반인가 | AiEO 보조 |
| **Y4. AI 안전성 회피 반응** | **AI가 "전문가와 상담하세요"라고 추천을 회피하는 비율** | **의료기기 전용** |

#### Y2의 이중 정의

- **이진형**: 추천됨(1) / 추천 안 됨(0) → 카이제곱, 로지스틱 회귀
- **연속형**: 추천 빈도 비율 (예: 15회 중 10회 = 0.67) → robustness check
- 두 분석 결론이 일치하면 robustness 입증

### 4.3 반복 실험 프로토콜

| 항목 | 설정 | 근거 |
|------|------|------|
| 반복 횟수 | 동일 쿼리-상품 조합당 **15~20회** | 빈도 기반 추천 확률 산출 |
| temperature | **default (0.7 또는 API 기본값)** | 실제 환경과의 괴리 방지. 0 고정 안 함. |
| 모델 버전 | 특정 버전 고정 (예: gpt-4o-2024-08-06) | 실험 재현성 |
| 순서 셔플 | 매 반복마다 **상품 제시 순서 랜덤 셔플** | position bias 통제 |
| 신뢰구간 | 이항 비율의 95% CI 함께 리포트 | 통계적 신뢰성 |
| 비용 관리 | 파일럿(20~30개 × 5회)으로 변동성 먼저 측정 | 반복 횟수 조정 근거 |

### 4.4 가설 및 메커니즘적 근거

| 가설 | 내용 | 메커니즘 |
|------|------|---------|
| H1 | HTML 포맷(TABLE/BULLET/PARAGRAPH)에 따라 파싱 정확도가 다르다 | TABLE은 LLM 토크나이저가 구조적 패턴으로 인식 |
| H2 | JSON-LD 유무/스키마 종류가 파싱 정확도에 영향 | MedicalDevice 스키마가 Product보다 의료기기에 최적화 |
| H3 | JSON-LD 효과가 HTML 포맷에 따라 다르다 (F1×F2 교호작용) | 이미 TABLE인 페이지에는 JSON-LD 추가 효과 작음 |
| H4 | 수치 구체성이 높을수록 파싱 정확도 높음 | "1200Hz"는 추출 대상 명확, "강력한"은 해석 필요 |
| H5 | 인증 정보가 명시적일수록 추천 확률 높음 | AI가 인증 정보를 신뢰 신호로 사용 |
| H6 | 인증 정보가 명시적일수록 AI 안전성 회피(Y4) 감소 | 인증 확인되면 AI가 "안심하고" 추천 |
| H7 | 임상 데이터 인용이 사용자 후기보다 추천에 효과적 | AI가 과학적 근거를 더 높게 가중 |
| H8 | Y1(파싱)이 높아도 Y2(선택)가 반드시 높지 않다 → **GEO ≠ AiEO 실증** | 읽는 것과 선택하는 것은 다른 인지 단계 |
| H9 | 경쟁 상황에서 구조화 포맷의 이점이 단독보다 크다 | 비교 평가 시 구조화된 정보가 비교 기준으로 사용되기 쉬움 |

### 4.5 분석 파이프라인

```
Phase 1: 가상 페이지 제작 + API 실험 실행 (데이터 수집)
    ↓
Phase 2: 개별 가설검정 (카이제곱, t-test) → 유의하지 않은 변수 탈락
    ↓
Phase 3: 선별된 변수를 로지스틱 회귀에 동시 투입
         → p-value, odds ratio로 변수 간 상대적 영향력 해석
         → 교호작용 확인 (특히 F1×F2, F4×F6)
    ↓
Phase 4: Real-world Validation
         → 바디닥터K 실제 페이지 + 경쟁사 페이지로 외적 타당성 검증
         → Before/After 측정 (페이지 개선 전후)
```

**핵심**: Phase 2~3은 같은 가상 데이터셋 사용. 분석 방법만 다름. 실제 데이터는 Phase 4에서만.

| JSON-LD 유무 + 필드 수 | 구조화 데이터 | 페이지 소스 파싱 |
| 인증 정보 존재/위치 | 의료기기 인증 명시 여부 | 텍스트 분석 |
| 수치 구체성 점수 | 정확한 수치 vs 모호한 표현 비율 | NLP |
| 텍스트 길이 | 제품 설명 글자수 | 크롤링 |
| 이미지 수 | 제품 이미지 개수 | 크롤링 |
| 리뷰 수 / 평점 | 리뷰 데이터 | 크롤링 |
| 서드파티 언급 빈도 | Reddit, 네이버 카페, 블로그 | 검색 크롤링 |
| 섹션 구분 명확성 | 스펙/설명/리뷰가 HTML로 명확 분리되는 정도 | HTML 분석 |
| 가격 | 판매가 | 크롤링 |

### 5.2 오픈엔드 vs 클로즈드셋 상관 검증

클로즈드셋 실험의 대리 지표 타당성을 검증:

1. 오픈엔드 쿼리("케겔운동기 추천해줘") → AI 자발적 언급 제품 목록 수집 (50회)
2. 클로즈드셋 쿼리("바디닥터K, 이지케이, 닥터케이 중 뭐가 좋아?") → 선택 빈도 (50회)
3. Spearman rank correlation 측정
4. 상관 높음 → 클로즈드셋 타당성 입증 / 낮음 → discovery vs evaluation 분리 분석

### 5.3 분석 파이프라인

```
Phase 1: 크롤링 → 피처 추출 → EDA → 가설 탐색
    ↓
Phase 2: ChatGPT API로 추천 결과 수집 → Y변수 생성 (15~20회 반복, 빈도 기반)
    ↓
Phase 3: ML 모델 구축
         - Baseline: 로지스틱 회귀
         - Main: XGBoost, Random Forest
         - 해석: SHAP 피처 기여도
    ↓
Phase 4: 오픈엔드 vs 클로즈드셋 상관 분석
    ↓
Phase 5: 버티컬 비교 (의료기기 vs 가글 추천 드라이버 차이)
    ↓
Phase 6: 바디닥터K SHAP 진단 리포트 생성
```

### 7.2 바디닥터K 제품 페이지 진단 체크리스트

| F2 | Product/MedicalDevice JSON-LD? | 없음 (추정) | MedicalDevice 스키마 추가 |

| F4 | 인증 정보 위치? | 하단 또는 없음 (추정) | 상단 눈에 띄게 |

## 9. 발표 전략

### 9.1 수업별 톤 분리

발표 초반 1분 내에 명확히:
> "산공통에서는 통제 실험 기반 인과 추론, 데마에서는 실제 데이터 기반 예측 모델링. 같은 주제이지만 데이터셋, 분석 방법, 목적이 모두 다릅니다."

수업 발표에서는 학술적 구조(연구 질문 → 가설 → 실험 설계 → 검증)에 집중. 창업 이야기는 질문 시에만.

### 9.2 발표 초반 필수 슬라이드

1. **실제 AI 쇼핑 에이전트 작동 구조**: 상품 DB를 받아서 비교 평가하는 구조 → 클로즈드셋 실험의 현실 타당성
2. **실험 프로토콜 요약**: 반복 횟수, temperature, position bias 셔플

### 9.3 "So What" 슬라이드

```
[Before] 바디닥터K 카테고리 추천율: 14.5%
         페이지 상태: PARAGRAPH, JSON-LD 없음, 수치 모호, 인증 미명시

              ↓ AiEO 진단 엔진 적용 ↓

[After]  바디닥터K 카테고리 추천율: ??% (목표 30%+)
         개선: TABLE + MedicalDevice JSON-LD + 수치 명시 + 인증 상세
```

### 9.4 예상 질문 대비

| 질문 | 대응 |
|------|------|
| 클로즈드셋이 현실을 반영하나? | 실제 AI 에이전트가 상품 DB를 받아서 평가하는 구조 = 현실이 이미 클로즈드셋 |
| LLM 응답이 매번 달라지는데? | 15~20회 반복 + 빈도 기반 확률 + 신뢰구간 |
| GPT 결과가 다른 LLM에도? | 프로젝트 범위는 GPT 기반, Phase 4에서 3개 플랫폼 교차 검증 |
| 사전학습 효과 vs 페이지 구조 효과? | 유명/무명 브랜드 동일 조건 비교로 간접 측정 |

## 11. 피드백 기반 설계 원칙 (2차 발표 피드백에서 도출)

이 프로젝트는 두 차례의 발표 피드백을 반영하여 설계되었다. 핵심 원칙:

1. **실험 vs 현실 괴리 해소**: 실제 AI 에이전트의 작동 구조(상품 DB 비교 평가)를 근거로 클로즈드셋 실험의 타당성을 설명한다.
2. **재현성 확보**: temperature default + 15~20회 반복 + 빈도 기반 확률로 LLM 응답 변동성을 통제한다.
3. **Position Bias 통제**: 매 반복마다 상품 순서 랜덤 셔플. 방법론에 명시.
4. **Y변수 이중 정의**: 이진(추천 여부) + 연속(추천 빈도 비율)으로 robustness 확보.
5. **버티컬 일반화 불가 인정**: 모든 버티컬에 동일 로직 적용하지 않음. 버티컬 간 차이 자체가 발견.
6. **발표 톤 분리**: 수업별로 학술적 구조 중심. 사업 비전은 질문 시에만.
7. **So What 강화**: 바디닥터K Before/After로 실용적 결론 실증.
