## 🔴 P0-1. 반복 호출을 **최소 n=20**으로 (현재 n=2)

### 현재 상태 (문제)
프롬프트 1개당 **2회** 실행 → 149 프롬프트 × 2 = 298 응답.

### 왜 문제인가 (정량)
비율 추정값의 **신뢰구간 폭**이 n에 따라 극단적으로 다름. Wilson 95% 신뢰구간(이항 분포 표준):

| n (호출 수) | p=50% 가정 CI 폭 | p=10% 가정 CI 폭 |
|-----------|----------------|----------------|
| 2         | ±50%           | ±39%           |
| 5         | ±35%           | ±24%           |
| **20**    | **±21%**       | **±14%**       |
| 30        | ±18%           | ±12%           |
| 50        | ±14%           | ±9%            |

**현재 대시보드의 "긍정 비율 6.67%"는 n=30 기준 95% CI가 [1.1%, 21.4%]** — "사실상 0~21% 사이 어디든 가능" 이라는 뜻. 지표로서 의미 없음.

### 바꿀 것
```python
# 현재
for prompt in prompts:
    for i in range(2):
        response = call_llm(prompt)

# 바꿀
N_REPEAT = 20  # 최소
for prompt in prompts:
    for i in range(N_REPEAT):
        response = call_llm(prompt, seed=42 + i)
```

### 비용 검증
- 149 프롬프트 × 20회 = 2,980 호출
- gpt-5.4-nano: $0.20/$1.25 per 1M token (입력/출력)
- 호출당 평균 ~2,000 input + 500 output 토큰 = **$0.001**
- **총 약 $3** (299 → 2,980 호출, +10배지만 비용은 +10배 수준이 아님)

→ 비용 걱정 없이 증가 가능.

## 🔴 P0-2. 대시보드 헤더에 **실행 메타정보** 표시

### 현재 상태
상단에 "감사 실행일"만 표시. 어떤 모델·어떤 설정으로 돌렸는지 **모름** → 재현 불가.

### 왜 문제인가
"감사 실행일 2026-04-22"와 "2026-05-01"이 다른 모델·다른 temperature로 돌면 **결과 비교가 무의미**. 대시보드가 "월간 트렌드"를 보여줘도 해석 불가.

### 바꿀 것
대시보드 상단에 강제 표기:

```
모델: gpt-5.4-2026-04-15
Temperature: 0.7 (API 기본값 유지)
Seed 범위: 42 ~ 61 (반복별 seed_base + repeat_index)
총 호출 수: 2,980
실행 일시: 2026-04-24 01:18 KST
캐시 히트: 0% (cold run)
```

### 바꿀 코드 (예시)
```tsx
// 대시보드 헤더 컴포넌트
<DashboardHeader>
  <MetaBadge label="모델" value={run.model_version} />
  <MetaBadge label="Temperature" value={run.temperature ?? "default"} />
  <MetaBadge label="Seed" value={`${run.seed_min}~${run.seed_max}`} />
  <MetaBadge label="총 호출" value={run.total_calls.toLocaleString()} />
  <MetaBadge label="캐시 히트" value={`${run.cache_hit_rate}%`} />
</DashboardHeader>
```

jsonl 응답 파일 스키마 (각 라인):
```json

  "run_id": "run_20260424_0118",
  "model_version": "gpt-5.4-2026-04-15",
```

## 🔴 P0-3. 모든 비율 지표에 **95% 신뢰구간 error bar**

### 현재 상태
"추천률 0.0%", "긍정 6.7%" 같은 점추정치만 표시. n이 얼마인지, 신뢰도가 얼마인지 시각적 단서 없음.

### 왜 문제인가
위 P0-1 표에서 봤듯 같은 "추천률 0.0%"도 n=2인지 n=148인지에 따라 전혀 다른 신뢰도.

### 바꿀 것
모든 비율 숫자 옆/위에 **Wilson 95% 신뢰구간 error bar**.

### Wilson 95% CI 계산 (언어 무관 공식)
```javascript
function wilsonCI95(successes, n) {
  if (n === 0) return [0, 1];
  const z = 1.96;
  const p = successes / n;
  const denom = 1 + z*z / n;
  const center = (p + z*z / (2*n)) / denom;
  const half = z * Math.sqrt(p*(1-p)/n + z*z/(4*n*n)) / denom;
  return [Math.max(0, center - half), Math.min(1, center + half)];
}
// 사용: const [low, high] = wilsonCI95(10, 30);
// → [0.178, 0.528]
```

### 시각 예 (CSS 또는 chart lib)
```
중립 질문 언급률:   [ 0.0% ] ··········|
                   실제 추정치        CI: 0% ~ 2.5%

브랜드 지명 긍정:   [ 6.7% ] ····|········|
                                CI: 1.1% ~ 21.4%
```

- 점추정치 진한 색, CI 구간 연한 색
- 호버 시 툴팁: `n=30, 95% CI [1.1%, 21.4%]`

### 왜 중요한가 (실무)
"긍정 6.7%"만 보면 "부정적이다" 판단 가능하지만, CI [1.1%, 21.4%]를 함께 보면 **"아직 판단하기 n이 부족함"** 이 바로 보임. 의사결정 오류 방지.

## 🟡 P1-1. 호출 간 순서 셔플(Position Bias) 명시

### 현재 상태
경쟁사 리스트를 AI에게 보여줄 때 순서가 매번 같을 수 있음. AI는 **리스트 첫 번째 항목을 우대하는 경향**(position bias) 있음. 증거:
- Wang et al. (2023) "Large Language Models are not Fair Evaluators"
- Zheng et al. (2023) "Judging LLM-as-a-Judge"

### 왜 문제인가
경쟁사 순서가 고정이면 "1번 자리에 놓인 브랜드"가 계속 유리 → 측정값이 실제 제품 품질이 아니라 순서 아티팩트.

### 바꿀 것
각 호출마다 **경쟁사 리스트 순서 랜덤 셔플** (seed 기반 재현 가능하게):

```python
import random
rng = random.Random(42 + repeat_index)  # seed 고정
order = rng.sample(all_competitors, k=len(all_competitors))
# 이 order를 프롬프트에 사용 + jsonl에 기록
```

jsonl 필드 `products_order: [...]` 필수. 분석할 때 순서별 편향 검정 가능.

## 🟡 P1-2. 쿼리 유형 서브분류 확장

### 현재 상태
4유형 (중립 / 브랜드 지명 / 경쟁사 대안 / 직접 비교). "중립" 148건이 한 덩어리.

### 왜 문제인가
"중립" 안에 **서로 다른 사용자 의도**가 섞여 있음:
- "가글 추천해줘" (카테고리 일반)
- "입냄새 때문에 고민인데 뭐 사야 돼?" (증상 기반)
- "1만원대 가글 뭐 좋아?" (가격 기반)
- "출근 전에 쓸 가글 뭐 좋아?" (유스케이스)

이 4가지는 **AI 응답 패턴이 달라야 정상**. 한 덩어리로 보면 평균화되어 insight 누락.

### 바꿀 것
"중립 148건"을 아래 **4개 서브카테고리로 라벨링**:

| 서브코드 | 의미 | 예시 |
|--------|------|------|
| CAT | 카테고리 일반 | "가글 추천해줘" |
| SYM | 증상 기반 | "입냄새 심한데 어떤 가글?" |
| PRC | 가격 기반 | "1만원대 가글 뭐 좋아?" |
| USE | 유스케이스 | "출근 전에 쓸 가글" |

### 구현
각 프롬프트에 `subcategory` 필드 추가. 대시보드에서 필터 드롭다운 or 탭.

```json
{
  "prompt_id": "NEUTRAL-001",
  "prompt_text": "구강청결제 처음 써보는데 뭐 사면 돼?",
  "category": "neutral",
  "subcategory": "USE"   // 신규
}
```

## 🟡 P1-3. 감성 분류기 스펙 공개

### 현재 상태
"긍정 6.67% / 중립 93.33% / 부정 0%" 결과만 표시. **어떤 모델·어떤 프롬프트로 분류했는지 불명**.

### 왜 문제인가
중립 93.33% 쏠림은 **분류기의 보수적 기본값** 편향일 가능성 매우 높음. 리스테린 같은 글로벌 브랜드가 AI 응답에서 "그냥 중립으로만 언급"되는 건 실제로는 "긍정적 권위 인용"일 수 있음.

### 바꿀 것
대시보드 어딘가(혹은 about 페이지)에 분류기 스펙 공개:
- 모델명 + 버전 (예: gpt-5.4-nano)
- 시스템 프롬프트 풀 텍스트
- few-shot 예시 (있다면)
- 검증 방법 (예: 100건 수동 라벨 대비 정확도 82%)

### 추가 권고
- 강제 3분류 대신 **4분류 (긍정/중립/부정/모호)** 도입
- 확신도(confidence) 0.7 미만은 "모호"로 분류 → 분류기 불확실성 시각화

## 🟡 P1-4. Raw 응답 jsonl 다운로드 지원

### 현재 상태
대시보드는 집계 결과만. 개별 응답 원문 확인 불가.

### 왜 문제인가
- 외부 검증 불가 (감사 투명성)
- 재분석 불가 (같은 raw로 다른 지표 산출)
- 이상치 추적 불가 ("왜 이 프롬프트만 100%인지" 원문 확인)

### 바꿀 것
각 감사 run마다 "Raw 데이터 다운로드 (.jsonl)" 버튼:
```
┌─────────────────────────────────┐
│ run_20260424_0118 (298 calls)   │
│ [다운로드 responses.jsonl 450KB] │
│ [다운로드 config.json]           │
└─────────────────────────────────┘
```

## 🟡 P1-5. 가격형·유스케이스·구매의도 쿼리 추가

### 현재 상태
4유형 중 "경쟁사 대안"과 "직접 비교" 외에는 **가격·유스케이스·구매의도** 쿼리 부재.

### 왜 필요
실제 사용자 쿼리 분포에는 이 3가지가 상당 비중. 누락하면 AI 행동의 중요한 축을 못 봄.

### 바꿀 것
각 유형 6개씩 총 18 프롬프트 추가:

**PRC (가격)**:
- "1만원대 가글 추천해줘"
- "프리미엄 가글 3만원대로 뭐가 좋아?"
- "마트에서 살 수 있는 5천원 이하 가글"
- "할인 자주 하는 가글 브랜드 뭐 있어?"
- "비싼 만큼 효과 있는 가글 있을까?"
- "코스트코에서 파는 대용량 가글 추천"

**USE (유스케이스)**:
- "출근 전에 쓰기 좋은 가글"
- "치과 치료 후에 쓸 수 있는 가글"
- "여행 갈 때 휴대하기 좋은 가글"
- "아이도 쓸 수 있는 순한 가글"
- "술 마신 다음 날 쓸 가글"
- "퇴근 후 긴 일과 마무리용 가글"

**DEC (구매의도, 단일 선택)**:
- "가글 딱 하나만 골라줘"
- "지금 바로 주문할 가글 추천 1개"
- "친구 선물용 가글 하나만 추천해줘"
- "지금 사면 안 후회할 가글 단 하나"
- "내일부터 쓸 건데 한 제품만 딱 알려줘"
- "가글 처음 사는 사람한테 딱 하나만 추천"

## 🟢 P2 (장기, 여유 있을 때)

### P2-1. 응답 캐시 적중률 표시
같은 (모델, 프롬프트, seed) 조합 중복 호출 → 캐시로 API 비용 절감. 대시보드 요약에 "이번 run 캐시 히트 12%" 표시.

구현:
```python
cache_key = sha256(f"{model}||{prompt}||{seed}||{temp}")
cached = load_cache(cache_key)
if cached:
    response = cached
    from_cache = True
else:
    response = call_llm(...)
    save_cache(cache_key, response)
```

### P2-2. 이전 run과의 diff 비교
"지난달 대비 추천률 +8%p" 같은 추이. 월간 재측정 필수.

### P2-3. 다중 모델 비교 탭
같은 프롬프트를 여러 모델 (gpt-5.4, Gemini 1.5 Pro, Claude 4.5 등)에 돌린 결과 나란히 비교. 답변 일관성 확인.

### P2-4. Bootstrap CI
n이 부족한 셀(예: 특정 경쟁사 × 특정 쿼리 유형 n<10)은 Wilson CI 대신 **1000회 bootstrap** 으로 더 robust한 CI.
