# 팀 네이밍 규칙 — 신규 팀원도 5분 안에 이해되도록

**작성**: 2026-04-30 (팀 피드백 반영)
**적용 범위**: 새 파일·새 코드·새 문서. 기존 파일은 그대로 (이름 바꾸면 참조 깨짐).
**목적**: "v5가 뭐예요?", "BRD가 뭐예요?", "F2가 뭐예요?" 라는 *반복되는 질문*을 없애는 것.

---

## 1. 왜 어려웠는가 (원인 진단)

팀원이 헷갈리는 패턴 3가지:

### 원인 1 — 버전 번호가 *의미 없음*
**예**: `queries_medical_v5.yaml`
- v5가 v4보다 뭐가 다른지 파일명만 봐선 모름
- 파일을 열어서 주석을 읽어야 비로소 차이를 앎
- *원래 작성자(Wayne·Claude)는 알지만 새 팀원은 모름*

### 원인 2 — 영어 약어가 *글로서리 부재*
**예**: BRD / CAT / SYM / CMP / COM / PRC / USE / DEC (쿼리 8유형)
- 산공통/데마 팀원이 이 코드를 처음 봤을 때 의미를 모름
- 코드는 *원작자의 머릿속*에서 자연스럽지만 *팀에게는 외계어*
- 한국어 풀이름 없이 "마스터 §4.4 참조"라고만 적힘 → 막힘

### 원인 3 — 코드 letter shortcut이 *맥락 없이 등장*
**예**: F1, F2, ..., F6 / H1, H2, ..., H15 / Y1, Y2, ..., Y4 / R1, R2, ..., R10 / L18, L36 / Phase A/B/C/D / B1, B2, B3
- 이 모든 코드가 *서로 다른* 분류 체계 (요인 / 가설 / 종속변수 / 룰 / 직교배열 / 페이즈)
- 같은 letter (B, F)가 다른 의미로 재활용되어 더 헷갈림
- Glossary에 정의가 분산되어 있거나 외부 문서로 redirect

### 핵심 통찰

**코드는 *원작자의 두뇌*에 효율적이지만 *팀의 두뇌*에는 비효율적**. 코드 자체를 없앨 필요는 없음 — 다만 *처음 보는 사람이 즉시 이해할 수 있는* 한국어 brigde가 필수.

## 2. 새 파일 작성 룰 5개

### Rule 1 — 파일명은 *의미 기반*, 버전 번호 X

❌ **나쁨**: `queries_medical_v5.yaml`
- "v5가 뭐?" 라는 질문 즉시 발생

✅ **좋음**: `queries_medical_네이버검색반영.yaml` 또는 `queries_medical_2026-04-30.yaml`
- *변경 사유*가 파일명에 보임 → 무엇이 다른지 즉시 이해

### Rule 2 — 버전 번호 대신 *의미 접미사*

새 파일이 기존 파일을 *대체*가 아니라 *별도 트랙*인 경우:
- `_원본` (preservation)
- `_초안` (draft)
- `_검수전` / `_검수후`
- `_검색반영` (specific change)
- `_경쟁사반영`
- `_최종`

날짜 접미사도 가능: `_2026-04-30`

❌ **나쁨**: `queries_medical_v5.yaml`
✅ **좋음**: `queries_medical_네이버검색반영.yaml`
✅ **좋음**: `bodydoctor_pdp_copy_초안.md` (현재는 `_draft`)

### Rule 3 — 영어 약어는 *데이터에만* 유지, 문서엔 한국어 우선

데이터 파일 (jsonl, csv, yaml의 식별자):
- BRD, F2, H10, R1 등 *그대로 유지* (분석 코드 호환성)
- 영어 약어는 *간결한 라벨*로서의 가치 있음

마크다운 문서:
- 처음 등장 시 *한국어 풀이름 + (영어 약어)* 로 표기
- 한 번 풀이름 적었으면 이후 약어 사용 OK

❌ **나쁨**: "BRD 쿼리에서 anchor mention 21%"
✅ **좋음**: "브랜드 지명(BRD) 쿼리에서 anchor mention 21%"

### Rule 4 — Glossary 의무 인덱싱

새 코드를 만들 때 *반드시*:
1. `docs/knowledge/glossary.md` 에 한국어 한 줄 정의 추가
2. 처음 사용한 문서에 *Glossary 링크* 또는 *인라인 풀이름*

코드를 글로서리에 없이 새로 도입하면 *팀의 헷갈림 부채*가 누적됨.

### Rule 5 — 한국어 파일명 OK

- macOS / Linux / Git 모두 UTF-8 한국어 파일명 지원
- 기존 `findings/AiEO_경쟁사_리서치.md` 같이 이미 사용 중
- 단 *코드 파일* (.py, .yaml 식별자, .json 키)은 ASCII 유지 — 도구 호환성

## 3. 기존 파일은 어떻게

**Don't rename** — `mv` 하면 코드·문서의 모든 참조가 깨짐.

대신:
- 매 파일 *상단 주석/frontmatter에* "이 v몇 의미 + 무엇이 차이인지" 명시 (이미 v5 yaml에 했음)
- `glossary.md` 의 "Versions" 섹션에 모든 v* 의미 표로 집결

전환은 *새 파일부터*. 기존 파일은 *그대로 두되 주석·헤더로 보충*.

## 4. 자주 헷갈리는 코드 → 한국어 cheat sheet

### 쿼리 유형 (8유형)

| 코드 | 한국어 풀이름 | 예시 |
|---|---|---|
| BRD | 브랜드 지명 (Brand-Named) | "바디닥터 효과 있나?" |

| SYM | 증상 기반 (Symptom-Based) | "산후 요실금에 뭐 좋아?" |
| CMP | 비교 (Comparative) | "바디닥터 vs 이지케이" |
| COM | 경쟁사 후 대안 (Competitor-Named) | "이지케이 말고 다른 거" |

## 6. 컨벤션 적용 우선순위

신규 작성 시 100% 적용:
- 새 파일 (오늘부터)
- 새 commit message
- 새 코드 docstring·주석

기존 자료는 *수정 시*에 점진 보강:
- 문서 수정 시 첫 등장 코드에 한국어 풀이름 추가
- 큰 리팩토링 때만 파일명 변경 검토 (참조 다 추적해서 갱신할 수 있을 때만)

---

## 7. 새 코드를 도입할 때

새 letter shortcut(F7, H16, Y5, R11 등)을 도입할 때:

1. **반드시 glossary.md 에 추가** (한 줄 정의 + 풀이름)
2. **이 문서 (팀_네이밍_규칙.md) §4 cheat sheet 갱신**
3. **첫 사용 문서·commit message에 *한국어 풀이름과 함께* 등장**

이 3개 의식이 *팀의 헷갈림 부채*를 누적시키지 않는 핵심 절차.
